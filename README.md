# 金融合規雷達 (Compliance Radar)

專為台灣金融業法遵人員設計的 AI 輔助自動化工具。自動監測金管會 RSS 公告，利用 Claude AI 進行深度分析，並自動產生包含「合規建議」與「電子郵件草稿」的通知，可將重要期限一鍵加入 Google 日曆。

## 核心功能與法遵管理價值

本工具不只是自動化程式，更是為金融法遵部門設計的**數位化法規遵循解決方案**，涵蓋完整的法規生命週期管理：

### 1. 法規辨識 (Regulatory Identification)

- **自動監測**：持續追蹤金管會最新法令函釋與草案預告 RSS，確保法規情報不漏接。
- **AI 深度摘要**：使用 Claude Sonnet 4.6 語義理解技術，自動提取法規重點、影響範圍與具體合規建議。
- **附件偵測**：自動檢測並標註公告中的 PDF/DOC/XLS 附件，提醒人員下載深入研讀。

### 2. 影響評估 (Compliance Assessment)

- **Google 日曆整合**：AI 自動提取生效日與意見截止日並內嵌一鍵加入連結，直覺化控管合規期限。不需額外授權，符合金融機構資安規範。
- **合規建議產製**：由 AI 根據公告內容預產初步的合規建議，縮短法遵人員分析研判時間。

### 3. 通報與追蹤 (Reporting & Traceability)

- **自動產製草稿**：產生可直接發送給業務部門的 HTML 郵件通報，含 `.eml` 附件，落實法規傳達機制。
- **HTML 歷史報告**：自動產生按月分類的網頁版合規報告，確保管理軌跡可追蹤、可查詢。
- **執行紀錄 (Audit Ready)**：`run_history.jsonl` 詳實記錄每次檢查狀態，可供內部稽核與監理機關查核佐證。

## 監測範圍

預設監測以下金管會 RSS 來源：

| 來源名稱 | 公告類型 | 說明 |
| -------- | -------- | ---- |
| 金管會最新法令函釋 | 函釋 | 已生效或即將生效的法令解釋、規範 |
| 金管會法規草案預告 | 草案 | 預告中的草案，含意見徵詢截止日 |

如需追蹤其他來源（如證期局、保險局專屬 RSS），可在 `config.json` 的 `rss_sources` 陣列中新增項目，格式如下：

```json
{
    "name": "來源名稱",
    "url": "RSS 網址",
    "type": "函釋"
}
```

## 安裝步驟

1. **複製專案**：

   ```bash
   git clone https://github.com/ChaosEmily/compliance-radar.git
   cd compliance-radar
   ```

2. **建立虛擬環境**：

   ```bash
   python -m venv .venv
   source .venv/bin/activate  # Windows 請用 .venv\Scripts\activate
   ```

3. **安裝依賴**：

   ```bash
   pip install -r requirements.txt
   ```

4. **初始化設定**：

   - 執行腳本，首次執行會自動產生 `config.json`：

     ```bash
     python setup.py
     ```

   - 開啟 `config.json` 填寫 Gmail 帳號、應用程式密碼與 Anthropic API 金鑰後，再次執行 `setup.py`。
   - 該腳本會自動在 **Windows 工作排程器** 中建立定時任務（預設每 10 分鐘檢查一次）。

## 如何使用

### 手動執行

```bash
python scraper.py
```

### 設定說明 (config.json)

| 欄位 | 說明 |
| ---- | ---- |
| `gmail_user` | 發信用 Gmail 帳號 |
| `gmail_app_password` | Gmail 應用程式密碼（非登入密碼，需至 Google 帳戶設定產生） |
| `recipient_email` | 法遵部門收件信箱 |
| `email_strategy` | `single_emails`（每則獨立寄出，預設）或 `digest`（彙整為一封） |
| `scrape_interval_minutes` | 排程執行間隔（分鐘），預設 10 |
| `anthropic_api_key` | Claude API 金鑰 |
| `rss_sources` | RSS 來源陣列，每筆含 `name`、`url`、`type` 三個欄位 |

## 資安與隱私保護

- **本地端執行優先**：所有敏感設定（API Key、郵件帳密）均儲存於本地 `config.json`，不入版控，不經由第三方伺服器處理。
- **數據最小化原則**：系統僅將「金管會公開公告內容」發送至 Claude 進行語義分析，符合金融業對數據隱私與資安控管的要求。
- **高安全性整合**：Google 日曆功能採用標準 URL Scheme 達成，無需與 Google 帳密連動或進行 API 授權。

### 通知頻率說明

`scrape_interval_minutes`（排程檢查頻率）與法遵人員實際收到信的頻率**不相同**：

- **排程檢查**：系統每隔指定分鐘（預設 10 分鐘）檢查一次金管會 RSS，比對是否有尚未處理的新公告。若無新公告則靜默結束，**不會寄信**。
- **Email 通知**：僅在發現新公告時才觸發。金管會通常於上班時間發布公告，因此實際收信頻率取決於公告發布節奏，而非排程間隔。

以預設設定為例：

| 情境 | 排程行為 | 法遵人員收信 |
| ---- | -------- | ------------ |
| 非上班時間，無新公告 | 每 10 分鐘檢查，靜默結束 | 不收信 |
| 上班時間，金管會發布 1 則函釋 | 最遲 10 分鐘內偵測到 | 收到 1 封通知 |
| 上班時間，同時發布 3 則公告 | 一次偵測到 3 則 | `single_emails`：3 封 / `digest`：1 封彙整 |
| 整天無新公告 | 全天靜默 | 不收信 |

如希望降低檢查頻率以減少 API 呼叫成本，可將 `scrape_interval_minutes` 調高（如 60 分鐘），代價是偵測延遲增加。

### 信件產出規則

#### 信件標題

系統根據公告類型與寄送模式自動產生標題：

| 寄送模式 | 標題格式 | 範例 |
| -------- | -------- | ---- |
| `single_emails` | `📜【函釋】AI 合規建議 - {公告標題}` | 📜【函釋】AI 合規建議 - 修正「銀行辦理衍生性金融商品業務應注意事項」 |
| `single_emails` | `📝【草案】AI 合規建議 - {公告標題}` | 📝【草案】AI 合規建議 - 預告修正「保險業內部控制及稽核制度實施辦法」 |
| `digest` | `【本日合規情報總覽】共有 N 則新法規異動` | 【本日合規情報總覽】共有 3 則新法規異動 |

- 函釋類公告前綴 📜，草案類公告前綴 📝
- 彙整模式下只有一個統一標題，個別草稿以 `.eml` 附件呈現

**`digest` 模式的寄送頻率**：

若金管會某天在上午跟下午，分別發出一則新公告，法遵人員會收到一封「彙整」信，還是兩封信？**兩封信**。

`digest` 模式的「彙整」是指同一次排程檢查中發現的多則公告合併為一封，而非「同一天的公告」合併。彙整信並非「每日定時」寄出，而是與 `single_emails` 相同 —— 每次排程檢查發現新公告時立即寄出。

以排程每 10 分鐘執行為例：

- **上午 10:05** 金管會發布 1 則 → 10:10 排程偵測到 → 寄出 **1 封**彙整信（含 1 則）
- **下午 14:25** 金管會再發布 1 則 → 14:30 排程偵測到 → 寄出 **第 2 封**彙整信（含 1 則）

只有同一批次偵測到 2 則以上時，`digest` 和 `single_emails` 才有差異。若每次只偵測到 1 則，兩種模式的收信體驗完全相同。若排程每 10 分鐘執行一次，而金管會在 14:00 與 14:30 各發布一批公告，法遵人員會收到 2 封彙整信，而非 1 封每日總覽。

#### 信件內容結構

每封通知信包含以下區塊（由上至下）：

| 區塊 | 說明 |
| ---- | ---- |
| **AI 綜合摘要** | 一句白話文摘要，快速理解公告重點。函釋為綠色區塊，草案為橘色區塊 |
| **附件提醒** | 若公告含 PDF/DOC 等附件，顯示黃色警示條 |
| **加入 Google 日曆** | AI 從公告中提取生效日或意見截止日時顯示。生效日為藍色按鈕，意見截止日為橘色按鈕 |
| **內部通知草稿** | AI 產生的正式通知草稿，可直接轉寄給業務部門。含建議主旨 |

彙整模式（`digest`）額外提供：

- 信件本文為所有公告的摘要總覽
- 每則公告的完整草稿以 `.eml` 附件方式夾帶，下載後可直接用郵件客戶端開啟發送
- `.eml` 草稿內含日曆邀請附件（`.ics`），收件人開啟後可直接加入行事曆

## 檔案結構

- `scraper.py`: 核心抓取、AI 分析與發信邏輯。
- `setup.py`: 初始化設定與排程註冊。
- `config.json`: 系統設定（執行時自動產生，不入版控）。
- `processed_announcements.json`: 已處理的公告紀錄，避免重複發信（不入版控）。
- `run_history.jsonl`: 每次執行的紀錄，含時間戳、處理筆數、成功/失敗狀態（不入版控）。
- `reports/`: 自動產生的 HTML 月報與索引頁。

## 資料保留

- HTML 月報存放於 `reports/` 目錄，系統啟動時會自動檢查超過 **5 年** 的報告並提醒歸檔。
- `run_history.jsonl` 持續追加，可定期歸檔或清理。
- 保留策略可依各公司內部文件管理辦法調整。

## 支援平台

- **Windows 10/11**：完整支援，含 Windows 工作排程器自動註冊與桌面通知。
- **Linux / macOS**：`scraper.py` 可正常執行，排程需手動設定 Crontab（`setup.py` 會提示指令）。

本工具產出之摘要與草稿僅供內部參考，不代表正式法律意見。所有法規異動請務必以主管機關官方公告及正式法規條文為準。
